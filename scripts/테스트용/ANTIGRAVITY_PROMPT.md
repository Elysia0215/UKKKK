# 정책 우선순위 매트릭스 개편 프롬프트

아래 내용을 그대로 안티그래비티의 에이전트 채팅에 붙여 넣는다. `policy_matrix_engine.py`도 프로젝트 루트에 함께 첨부한다.

---

당신은 시니어 프로덕트 디자이너이자 데이터 엔지니어다. 현재 프로젝트의 정책 우선순위 진단 화면을 실제 의사결정에 사용할 수 있도록 개편하라.

## 작업 원칙

1. 먼저 프로젝트 전체 구조, 실행 방법, 데이터 모델, API, 현재 매트릭스 화면을 찾아 분석하라.
2. 기존 디자인 시스템, 색상 토큰, 컴포넌트, 폰트, 라우팅 방식을 유지하라. 전면 재작성하지 말고 필요한 부분만 수정하라.
3. `policy_matrix_engine.py`의 결과 스키마와 점수 정의를 기준으로 백엔드/API 및 화면을 연결하라.
4. 가짜 데이터로 기존 실제 데이터 경로를 덮어쓰지 마라. 실제 연동이 불가능한 경우에만 명시적인 fixture 파일을 별도로 만들고, 교체 지점을 주석과 README에 남겨라.
5. 변경 전 관련 파일을 확인하고, 변경 후 lint·typecheck·test·build를 실행하라. 실패하면 원인을 수정한 후 다시 검증하라.
6. 기존 기능과 모바일/데스크톱 레이아웃을 깨뜨리지 마라.

## 문제 정의

현재 화면은 시민 제안, 민원, 기존 정책, 뉴스 건수를 대분류별로 합산한 뒤 하나의 우선점수와 추천 액션으로 압축한다. 이 때문에 서로 다른 문제가 한 분류 안에 섞이고, 여러 행이 100점으로 포화되며, 담당자가 무엇을 고쳐야 하는지 알기 어렵다.

대분류는 탐색용 폴더로만 사용하고 실제 평가 단위는 구체적인 `문제 클러스터`로 변경하라.

정보 구조는 다음과 같다.

`대분류 → 문제 클러스터 → 대표 시민 요구 및 근거 원문`

예시:

- 의료·건강·심리 지원
  - 임신·출산 이용기준
  - 정신건강·상담
  - 의료비·경제 부담
  - 의료 접근성

## 진단 모델

각 문제 클러스터마다 다음 5개 축을 0~100으로 별도 표시하라.

- 수요 강도 `demand`: 중복 제거 후 제안 규모와 상대적 빈도
- 정책 공백 `policy_gap`: 미해결 정도 60% + 기존 정책 미포괄 정도 40%
- 사회적 시급성 `urgency`: 안전·생명·취약계층·시간 민감도
- 실행 가능성 `feasibility`: 권한, 비용, 소요 기간, 행정 난이도
- 근거 신뢰도 `evidence_confidence`: 출처 다양성과 필드 완전성

참고용 종합 우선점수는 아래 계산식만 사용한다.

`priority_score = demand×0.30 + policy_gap×0.25 + urgency×0.25 + feasibility×0.10 + evidence_confidence×0.10`

종합점수만으로 결론을 내리지 말고 5개 축을 항상 함께 보여줘라. 점수는 소수점 없이 반올림하고 0~100 범위를 벗어나지 않게 한다.

상태 라벨 규칙:

- 시급성 ≥75 및 정책 공백 ≥65: `즉시 검토`
- 정책 공백 ≥65 및 실행 가능성 ≥60: `제도 개선`
- 실행 가능성 ≥70 및 정책 공백 ≥40: `빠른 개선`
- 나머지: `모니터링`

## 메인 화면 요구사항

기존의 대분류별 한 행 구조를 문제 클러스터 중심 구조로 바꾼다.

1. 상단 필터
   - 분석 기간
   - 대분류
   - 상태 라벨
   - 담당 부서
   - 최소 근거 신뢰도
   - 검색
2. 요약 영역
   - 즉시 검토 클러스터 수
   - 새로 급상승한 클러스터 수
   - 근거 부족 클러스터 수
   - 이번 기간 해결/완화된 클러스터 수
3. 매트릭스
   - 기본 X축: 정책 공백
   - 기본 Y축: 사회적 시급성
   - 버블 크기: 수요 강도
   - 버블 투명도 또는 테두리: 근거 신뢰도
   - 색상: 상태 라벨
   - 접근성을 위해 색상만으로 상태를 구분하지 말고 텍스트/아이콘/패턴을 함께 사용
4. 테이블
   - 대분류, 문제 클러스터, 수요, 정책 공백, 시급성, 실행 가능성, 근거 신뢰도, 상태, 담당 부서
   - 기본 정렬은 우선점수 내림차순
   - 100점 배지를 과도하게 강조하지 말고 5개 축의 미니 바 또는 레이더가 아닌 수평 막대를 사용
   - 대분류 행은 펼침/접힘 그룹으로만 사용

## 상세 패널 요구사항

문제 클러스터를 클릭하면 우측 패널을 연다.

- 제목: 대분류 / 문제 클러스터
- 상태 라벨과 참고용 우선점수
- 5개 진단 축의 값, 산출 근거, 데이터 완전성
- 대표 시민 요구 3~5개와 원문 보기
- 출처별 건수, 중복 제거 전후 건수, 최근 증감
- 연결된 기존 정책과 해결되지 않은 조건
- 추천 액션을 `즉시 할 일 / 추가 확인 / 중장기 검토`로 분리
- 담당 부서, 담당자, 보고서 생성 버튼
- 근거 신뢰도가 50 미만이면 결론 대신 `근거 추가 수집 필요` 경고를 표시

## 데이터/API 연결

프로젝트의 실제 데이터 필드를 아래 표준 필드에 매핑하라.

- `id`: 원문 고유 ID
- `category`: 대분류
- `title`, `content`: 제목과 내용
- `source`: 시민제안/민원/뉴스/현장조사 등
- `created_at`: ISO 날짜
- `unresolved`, `urgency`, `feasibility`, `policy_coverage`: 있으면 0~100 값 사용, 없으면 엔진 추정값 사용

`policy_matrix_engine.py`를 그대로 호출하거나 동일 로직을 현재 백엔드 언어로 이식할 수 있다. 이식할 경우 계산식, 상태 규칙, 출력 필드명을 동일하게 유지하고 Python 원본과 같은 fixture를 넣었을 때 결과가 일치하는 테스트를 작성하라.

필수 결과 필드:

`category, cluster, item_count, source_count, demand, policy_gap, urgency, feasibility, evidence_confidence, priority_score, status, recommended_action, representative_titles`

## 품질 및 예외 처리

- 제목과 본문이 동일한 단순 중복은 집계 전에 제거한다.
- 날짜, 필수 필드, 점수 범위를 검증한다.
- 데이터가 없으면 0점처럼 보이게 하지 말고 `데이터 없음`을 표시한다.
- 근거가 부족한 추정 점수는 UI에서 `추정` 배지를 표시한다.
- 로딩, 빈 상태, 오류, 긴 텍스트, 대량 데이터, 모바일 화면을 처리한다.
- 한국어 문구를 자연스럽게 유지한다.
- 색 대비와 키보드 포커스 등 WCAG AA 수준의 접근성을 고려한다.

## 테스트 시나리오

최소한 다음을 검증하라.

1. 한 대분류에 서로 다른 요구가 들어오면 여러 문제 클러스터로 나뉜다.
2. 중복 제안은 수요 건수에 두 번 포함되지 않는다.
3. 입력 점수는 0~100으로 제한된다.
4. 상태 라벨의 경계값 64/65, 74/75가 정확히 동작한다.
5. 근거 신뢰도 50 미만에서 경고가 보인다.
6. 필터·정렬·상세 패널이 키보드로 동작한다.
7. 기존 페이지 및 API 회귀 테스트가 통과한다.

id,category,title,content,source,created_at
